# Neural Extractor V3.0.3 reliability design

## Authentication and retry order

Public YouTube URLs start with `auth=none`, even when `cookies.txt` is available.
The bounded state-driven order is:

1. No cookies, default YouTube client, requested media selector.
2. One fresh no-cookie subprocess retry after a transient network failure, HTTP 403,
   inactivity timeout, or total timeout.
3. One alternative no-cookie client only after bounded discovery reports concrete media
   format IDs. A selector such as `best` is never invented after discovery.
4. `cookies.txt` only after yt-dlp explicitly reports authentication is required.
5. Detected browser profiles in deterministic Chrome, Edge, Brave, Firefox order only
   while authenticated fallback remains justified.

Each authenticated provider is tried at most once per job. A locked browser database,
DPAPI/decryption failure, or extraction failure disables that provider for the rest of
the job. Generic HTTP 403 does not enable authenticated fallback.

## Attempt isolation and timeouts

Every yt-dlp attempt runs in a new internal Neural Extractor worker subprocess. This
isolates yt-dlp HTTP sessions, extraction state, reader threads, Node, and FFmpeg from
the GUI and from later retries.

Defaults:

- Inactivity timeout: 300 seconds without meaningful stdout, stderr, phase, or progress.
- Total attempt timeout: 21,600 seconds (6 hours).
- GUI activity heartbeat: 15 seconds.
- Graceful termination wait: 3 seconds.
- Forced tree-kill wait: 3 seconds.

The limits can be configured before startup with:

- `NEURAL_EXTRACTOR_YTDLP_INACTIVITY_TIMEOUT_SECONDS`
- `NEURAL_EXTRACTOR_YTDLP_ATTEMPT_TOTAL_TIMEOUT_SECONDS`
- `NEURAL_EXTRACTOR_YTDLP_STATUS_HEARTBEAT_SECONDS`
- `NEURAL_EXTRACTOR_YTDLP_TERMINATION_GRACE_SECONDS`

Partial `.part` files are preserved so yt-dlp's normal resume behavior remains usable.
Completed files are never deleted during cancellation or recovery.

## Windows process-tree termination

Neural Extractor starts the internal worker with `shell=False`,
`CREATE_NEW_PROCESS_GROUP`, and no PowerShell or cmd wrapper. It records the exact child
PID, executable, process creation identity, owner PID, and owner creation identity. It
never records the URL, yt-dlp options, cookie path, or stdin payload.

Cancellation or timeout validates that exact PID and creation identity, requests a
graceful `taskkill /PID <pid> /T`, waits for the bounded grace period, and escalates to
`taskkill /PID <pid> /T /F` only if the owned tree remains. This terminates descendant
FFmpeg, Node, and helper processes without matching or killing processes by name.

If Neural Extractor crashes, the next launch scans only Neural Extractor ownership
records. Recovery runs only when the original owner is gone and the child PID still has
the exact recorded creation identity. PID reuse, an active original owner, or the current
GUI PID prevents termination.

## Owner field test

1. Place a valid `cookies.txt` beside or select it in Neural Extractor.
2. Start V3.0.3 and download an ordinary public video.
3. Confirm the Activity Log says `cookies.txt available` and the first profile says
   `auth=none`.
4. Confirm it does not say `Authenticating with cookies.txt` for that successful public
   download.
5. Start another legal test download and press Stop while it is active.
6. Confirm the row immediately changes to `Cancelling`, then `Cancelled`, and the queue
   returns to an idle/usable state without a reboot.
7. Start another download immediately and confirm it can complete.
8. If a legal authentication-required video is available, test it and confirm
   `cookies.txt` is attempted only after the explicit authentication failure.
9. If `cookies.txt` is rejected, confirm it is not repeated and each browser provider is
   attempted no more than once.
10. From an installed V3.0.2 build, check for updates after V3.0.3 is eventually published
    and confirm the existing manifest, SHA-256, detached installer, startup confirmation,
    backup, and rollback flow recognizes V3.0.3 as newer.

No PO Token is collected, generated, hardcoded, or fetched from a third-party service.
When yt-dlp explicitly requires one, V3.0.3 reports that separate limitation and stops.
