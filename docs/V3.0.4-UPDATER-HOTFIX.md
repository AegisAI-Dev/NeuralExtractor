# Neural Extractor V3.0.4 updater handoff hotfix

## Purpose

V3.0.4 repairs the Windows self-updater transaction handoff used by the packaged
PyInstaller application. It preserves the V3.0.3 downloader, verification,
backup, startup-confirmation, and rollback behavior while correcting false
concurrent-updater detection.

## Confirmed root cause

V3.0.2 and V3.0.3 copied the running one-file executable to a detached updater
helper and launched that copy. The GUI then recorded the PID returned by
`subprocess.Popen` as the installation-lock owner.

A PyInstaller one-file executable has a bootloader process and a separate child
process that executes the Python application. The PID returned to the GUI was
the bootloader PID, while updater-helper code ran under the child PID. The helper
accepted only the original GUI PID or its own runtime PID. It therefore mistook
its bootloader PID for an unrelated updater and emitted:

```text
The update could not be applied safely.
Another updater process owns this installation.
```

Closing Neural Extractor and retrying could not correct the mismatch because
every packaged helper launch recreated the same bootloader/runtime PID split.

## V3.0.4 transaction model

The repaired updater treats the update as one explicit transaction and keeps
download staging, installation ownership, and startup confirmation as separate
responsibilities. Its state progression is:

```text
checking -> downloading -> downloaded -> verified -> helper_prepared
-> handoff_pending -> handed_off -> waiting_for_parent_exit -> backing_up
-> replacing -> launching -> awaiting_confirmation -> confirmed
```

Handled failures transition to `failed` before replacement or to
`rolling_back` and then `rolled_back` after replacement.

The handoff follows these rules:

1. The GUI creates one random transaction ID and persists the normalized target
   executable, its own PID and process-creation identity, and the current state.
2. The downloaded executable must pass manifest size and SHA-256 verification
   before helper preparation.
3. The GUI persists `handoff_pending` and releases installation ownership before
   launching the detached helper.
4. The helper receives the same transaction reference, validates the transaction
   and target, then atomically records its own runtime PID and creation identity.
   It never treats the PyInstaller bootloader PID as the Python helper owner.
5. Only ownership for the normalized target executable is exclusive. A different
   live transaction for that target is rejected, while a different installation
   path is independent.
6. The helper waits for the exact GUI process identity to exit, creates and
   verifies a backup, replaces the executable, and launches the new version.
7. The new version must produce the transaction-bound startup confirmation within
   the bounded timeout. Only confirmed startup permits backup and ownership
   cleanup. A failed confirmation triggers verified rollback.

PID checks always include process-creation identity because Windows can reuse a
PID. A missing process or a creation-identity mismatch is stale state, not proof
of a live competing updater. Stale recovery remains target-scoped and does not
delete a validated live competitor's ownership.

Ownership and diagnostic records contain only sanitized updater metadata. They
must not contain download URLs, cookies, credentials, commands, or environment
data, and no secret is passed through command-line arguments.

## Compatibility and one-time manual installation

The faulty helper code is part of the already installed V3.0.2 and V3.0.3
executables. Downloading V3.0.4 does not replace that old helper before it tries
to apply the update. Automatic installation from those versions is therefore not
promised and may reproduce the ownership error.

V3.0.2 and V3.0.3 users should perform one manual installation:

1. Finish or cancel active downloads and close Neural Extractor.
2. Open the official `AegisAI-Dev/NeuralExtractor` GitHub Release for V3.0.4.
3. Download `NeuralExtractorV3-3.0.4-windows-x64.exe` only from that release.
4. Optionally compare its SHA-256 with the published `.sha256` file and manifest.
5. Replace the previous executable or place V3.0.4 in the intended installation
   directory, then launch it and confirm that it reports version `3.0.4`.

Do not overwrite or modify the published V3.0.2 or V3.0.3 releases or tags.

## Future update expectation

Once V3.0.4 is installed manually, its repaired updater is intended to provide
the automatic update path to V3.0.5 and later compatible releases. That path
must still be validated with a real V3.0.4-to-V3.0.5 field update before it is
described as proven on an owner's PC.

V3.0.4 release manifests declare `minimum_updater_version` as `3.0.4`. Older
updaters must use the manual download path instead of being presented as a
supported automatic installation route.

## Security and signing limitations

Release packages remain protected by strict filename and manifest validation,
bounded size checks, SHA-256 verification, backup-before-replacement,
transaction-bound startup confirmation, and rollback. The GitHub manifest is
not independently signed, so these checks do not protect against compromise of
the official repository, release credentials, or build pipeline.

The Windows executable is not Authenticode-signed. Windows SmartScreen may show
a warning. Users should download Neural Extractor only from the official
`AegisAI-Dev/NeuralExtractor` GitHub Release and verify the published checksum
when possible.
